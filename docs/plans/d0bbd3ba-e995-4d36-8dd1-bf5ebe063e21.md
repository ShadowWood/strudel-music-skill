Complexity: Medium

## Goal
Reimplement the `strudel-music` skill so users can describe music in natural language and have the system produce a running Strudel REPL in a browser within one command, with the agent only ever editing a single `pattern.js` file.

## Context
- The project ships a Vite-based local player that embeds the official `strudel.cc` REPL inside an `<iframe>`. The pattern code lives in `templates/src/pattern.js`, is base64-encoded with `TextEncoder`, and pushed into the iframe as a URL hash: `https://strudel.cc/?autoplay=1#<base64>`. Vite HMR calls a hook in `templates/src/main.js` whenever `pattern.js` changes.
- A previous attempt to use the `@strudel/web` npm package locally was abandoned: the `evalScope` exposed by `evaluate()` is not equivalent to the strudel.cc REPL context, missing `setcpm`, `$:`, and a working `prebake` for drum samples. The iframe approach (Approach B) is the verified, working architecture and must be the basis of this implementation.
- The agent-facing documentation (`SKILL.md`, `reference.md`, `genres.md`, `examples.md`) is the entire "brain" of the skill. It must teach the agent how to map natural language into Strudel code, warn about the `arrange()` + top-level `let` trap, and document the strict `pattern.js` export template.
- The runtime must work with no internet access during skill installation; an `install.sh` is provided that detects supported agents (Cursor, Claude Code, Hermes, Windsurf, OpenClaw, Xcode) and supports both symlink and copy install modes, plus global vs. project scope.
- All text in the project (code, comments, docs, scripts, user-facing strings) must be English only.

## Assumptions
- The target system has Node.js â‰Ą 18 available on `PATH`, plus `python3` for the sample downloader's JSON parsing. The init script will verify and abort with a clear message otherwise.
- The implementer has network access during development to validate pattern URLs against `https://strudel.cc/`. Offline operation is an explicit out-of-scope.
- Sample audio sources remain hosted at `https://raw.githubusercontent.com/geikha/tidal-drum-machines/main/machines/...` with the manifest at `https://codeberg.org/uzu/strudel/raw/branch/main/website/public/tidal-drum-machines.json`. If these URLs change, `download-samples.sh` is the only file that needs updating.
- The user accepts that after the very first HMR reload they may need to click the play button inside the iframe once (browser autoplay policy); the spec's "Known Limitations" section makes this explicit.

## Implementation Notes

### Sequencing rationale
Tasks are ordered so each step is independently verifiable and produces a runnable artifact. The order is:

1. **Project scaffolding** â€” git init, `.gitignore`, the directory skeleton, and the empty `package.json` shells.
2. **Player template** â€” Vite + `index.html` + `main.js` + `pattern.js`. This is the only part the user can actually run, so it must come first as a runnable substrate.
3. **Agent-facing documentation** â€” `SKILL.md` (the agent's contract) â†’ `reference.md` (syntax + NL mapping) â†’ `genres.md` (genre templates) â†’ `examples.md` (paired NLâ†’code). All four reference each other and must be written in one pass to keep terminology consistent.
4. **Installer scripts** â€” `install.sh` (agent discovery) â†’ `scripts/init.sh` (player init) â†’ `scripts/download-samples.sh` (offline sample prep).
5. **Verification harness** â€” `templates/test-cdn.html` and a small `vitest` smoke test for `encodePattern`.
6. **End-to-end smoke test** â€” actually run the Vite server and confirm `HTTP 200` from `http://localhost:5173/`.

### Key technical decisions (frozen by the spec)
- **No `@strudel/*` packages anywhere.** The iframe hash approach is the architecture. Any reviewer suggestion to "use `@strudel/web` for offline" is rejected â€” see Â§3 of the spec for the bug history.
- **`encodePattern()` must use `TextEncoder` then a byte loop then `btoa`.** A direct `btoa(code)` throws on any non-Latin-1 character (e.g. Chinese punctuation in a user prompt). The harness unit test in `encodePattern.test.js` will assert the function returns the same base64 as a hand-computed reference value for a known input like `"setcpm(90/4)\n$: sound(\"bd\")"`.
- **No `sandbox` on the iframe.** Sandboxed iframes block the WebAudio API and break strudel.cc. The verification matrix explicitly asserts the absence of `sandbox`.
- **`arrange()` must be inlined.** `reference.md` and `examples.md` both include a `// WRONG` and `// CORRECT` example. The skill explicitly forbids top-level `let pattern = ...` bindings.
- **Sample downloader must URL-encode spaces.** `Closed Hat.wav` must become `Closed%20Hat.wav` in the URL. The script uses `python3 -c "import sys, urllib.parse, json; ..."` rather than shell `sed` to avoid quoting bugs.
- **Idempotent init.** `scripts/init.sh` detects an existing `${HOME}/strudel-music/` and prompts before overwriting. `npm install` is skipped if `node_modules/` is already populated and `package.json` is unchanged (check via `sha256sum` comparison).

### Risk controls
- **Risk: an agent installs the skill but their tool has no shell access.** Mitigation: `install.sh` is plain POSIX bash; `init.sh` is also POSIX bash; both have clear stdout messages.
- **Risk: the strudel.cc REPL changes its hash-decode format.** Mitigation: the only file the agent ever writes is `pattern.js` (a plain string). If strudel.cc changes, the fix is in `templates/src/main.js` â€” a single 50-line file.
- **Risk: a contributor reintroduces `@strudel/web` and starts adding COEP/COOP headers.** Mitigation: `templates/package.json` has an explicit `// DO NOT ADD @strudel/* PACKAGES` comment, and the AC explicitly greps for `@strudel` and fails the build if any match.
- **Risk: `download-samples.sh` is run on a host with no network.** Mitigation: the script prints a clear "manifest download failed, skipping samples" message and exits 0 so `init.sh` continues.
- **Risk: HMR reloads cause iframe re-render with no audio.** Mitigation: `SKILL.md` documents this as a known limitation; users click â–¶ once after the first load.

## Out of Scope
- Running Strudel fully offline using `@strudel/repl` (would reintroduce all of Approach A's bugs and require COEP/COOP header configuration â€” explicitly rejected by Â§12 of the spec).
- A standalone GUI editor for patterns (the agent edits `pattern.js` in any editor the user prefers).
- A pattern versioning or history system (the user is expected to use git for that).
- A web-based chat UI for natural-language input (the agent reads `SKILL.md` directly; no chat shell is built into the player).
- Internationalization. All project text is English only, per the language requirement.
- Automatic agent-side detection that the user is in a coding session (the agent's own invocation decides when to load `SKILL.md`).
- Sample bank curation beyond what the upstream `tidal-drum-machines.json` manifest exposes.
- PostMessage-based communication with the iframe (the URL-hash approach is sufficient and is what the spec verifies).
